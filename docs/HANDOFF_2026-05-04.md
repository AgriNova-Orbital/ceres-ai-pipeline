## Project Purpose

Ceres AI Pipeline is an AgriNova-Orbital wheat rust risk prediction platform using Sentinel-2 time-series data, Google Earth Engine/Drive ingestion, staged GeoTIFF-to-NPZ dataset generation, CNN-LSTM training, threshold-oriented evaluation, and a WebUI control panel.

Primary user flow:

1. Export/download weekly Sentinel-2 risk rasters.
2. Normalize and inventory raw GeoTIFFs.
3. Build staged datasets at L1/L2/L4.
4. Run staged training matrix experiments.
5. Evaluate checkpoints and select best model/threshold.
6. Monitor and operate through the WebUI or CLI.

## Main Runtime Architecture

- `frontend/`: Next.js frontend, App Router, Tailwind CSS, default external dev port `3002`
- `apps/`: Flask/Gunicorn API and WebUI layer, default API/WebUI port `5055`
- `modules/jobs/`: RQ worker and background task definitions
- `modules/services/`: service-layer wrappers for dataset build, inventory, training matrix, evaluation
- `scripts/`: CLI entrypoints for direct operational execution and debugging
- `state/`: SQLite runtime state, ignored by git
- `data/`: raw and staged data, ignored by git
- `runs/`: training/evaluation outputs, ignored by git
- `reports/`: inventory/evaluation reports, ignored by git

Main request flow:

1. Frontend submits JSON to Flask API.
2. Flask validates input and enqueues RQ jobs.
3. Redis stores queue and job metadata.
4. Worker executes tasks and updates `job.meta`.
5. Frontend polls `/api/jobs` and `/api/jobs/<id>` for status.

## API Tools Refactor Completed

PR `#11` merged the API tools refactor into `main`.

Merged feature branch:

```text
feature/api-tools-refactor
```

Primary changed files:

- `apps/api_runs.py`
- `apps/templates/wheat_risk_webui.html`
- `apps/wheat_risk_webui.py`
- `modules/jobs/tasks.py`
- `tests/test_api_runs.py`
- `tests/test_job_worker.py`
- `tests/test_webui_enqueues_jobs.py`
- `tests/test_wheat_risk_webui.py`

Purpose:

- Replace generic script-command execution in `/api/run/*` endpoints with structured API-tool-style task calls.
- Keep RQ async execution, but pass typed/validated payloads to dedicated tasks.
- Reduce shell/subprocess command construction at API boundary.

Implemented:

- `/api/run/downloader`
  - `preview_export`, `run_export`, `download_all` enqueue `task_export_weekly_risk_rasters`
  - `refresh_inventory` enqueues `task_run_inventory`
  - `run_export`/`download_all` require `drive_folder`
- `/api/run/build` enqueues `task_build_dataset`
- `/api/run/train` enqueues `task_run_matrix`
  - canonical actions: `dry_run`, `run_matrix`
  - legacy `execute_train` aliases to `run_matrix`
- `/api/run/eval` enqueues `task_run_eval`
- WebUI downloader no longer enqueues `task_run_script_for_user` for export preview/run
- Added `task_export_weekly_risk_rasters` in `modules/jobs/tasks.py`
  - calls `scripts.export_weekly_risk_rasters.main(argv)` in-process
  - injects user OAuth token through `GOOGLE_OAUTH_TOKEN_JSON` when available

Important caveat:

- `task_run_script_for_user` still exists for legacy compatibility, but the API tools refactor reduces new API endpoint dependence on it.
